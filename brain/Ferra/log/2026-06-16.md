# 2026-06-16 — Migración a Cloudflare Pages + páginas legales

## Resumen

Migración completa de `coming-soon-mailer` (Worker monolítico) a Cloudflare Pages con HTML estático + Pages Function. Creación de páginas legales bilingües (privacidad y términos en EN+ES). Sitio en producción en https://ferra.es.

## Trabajo realizado

### Investigación
- Inspeccionado el Worker `coming-soon-mailer` (route `ferra.es/`, bindings: KV `SUBSCRIBERS`, secrets `TURNSTILE_SECRET` + `TURNSTILE_SITEKEY`)
- Verificado que el account Cloudflare (`f07baa80011ef9da206ccbcbd33ca27a`) contiene la zona `ferra.es` (`5247fb50190e6397c2141f51001c975c`)
- Confirmado que el repo `DaybyDay-csv/Ferra-Mailer` existía pero estaba vacío

### Decisiones
- Migración completa (no coexistencia Worker + Pages en el mismo dominio)
- Páginas legales en formato side-by-side columns EN+ES (no rutas separadas por idioma)
- Footer con 4 enlaces: Home, Privacy, Terms, Contact (mailto:admin@ferra.es)
- Copy GDPR-grade con LOPDGDD española
- Entidad: FERRA, jurisdicción española

### Implementación
1. Creado `~/Developer/Ferra-Mailer/` con estructura:
   - `index.html` (landing con formulario + footer de 4 enlaces)
   - `privacy.html` (EN+ES columnas)
   - `terms.html` (EN+ES columnas)
   - `functions/api/subscribe.js` (Pages Function: Turnstile verify + KV write)
   - `_headers` (security headers)
   - `wrangler.jsonc` (config Pages con KV binding)
   - `README.md` y `.gitignore`
2. Sitekey Turnstile hardcoded en `index.html` (público por diseño)
3. HTML validado con parser: balance de tags correcto en los 3 archivos
4. Sintaxis JS validada con `node --check`
5. JSONC validado con parser

### Git
- `git init -b main` en `~/Developer/Ferra-Mailer`
- Commit inicial con todos los archivos
- Push a `github.com/DaybyDay-csv/Ferra-Mailer` usando PAT con método seguro (credencial efímero, limpiado después)

### Cloudflare
1. Creado Pages project `ferra-mailer` (subdomain: `ferra-mailer.pages.dev`)
2. Deploy inicial via `wrangler pages deploy` (5 archivos + Functions bundle)
3. Verificado en `ferra-mailer.pages.dev`: 3 páginas en 200, footer con 4 enlaces
4. Set `TURNSTILE_SECRET` como Pages secret
5. Mailer Function testada: `invalid_email`, `spam` (honeypot), `turnstile` (token inválido) — todos los paths funcionan
6. Attached `ferra.es` como custom domain (status inicial: pending)
7. **BLOQUEO ENCONTRADO:** El Worker seguía sirviendo `ferra.es/` aunque todas las rutas estuvieran borradas via API. El OAuth token no podía leer DNS/page rules/rulesets de la zona
8. **Resuelto por el usuario:** Usuario eliminó el Worker `coming-soon-mailer` del dashboard
9. Cutover completado: `ferra.es` ahora sirve Pages

### Verificación final (post-cutover)

- `https://ferra.es/privacy` → HTTP 200, 15149 bytes ✅
- `https://ferra.es/terms` → HTTP 200, 16612 bytes ✅
- `https://ferra.es/api/subscribe` (GET) → HTTP 405 ✅
- `https://ferra-mailer.pages.dev/` → HTTP 200 ✅
- Headers de seguridad aplicados: X-Frame-Options, HSTS, Permissions-Policy, Referrer-Policy, X-Content-Type-Options ✅
- Turnstile verify funcionando (devuelve `turnstile` con token fake, no `turnstile_unconfigured`)

## Problemas encontrados

1. **Cloudflare OAuth token limitado:** El token OAuth de wrangler no podía leer DNS records, page rules, ni rulesets de la zona (errores 10000/9106). Esto bloqueó la inspección de reglas ocultas que mantenían al Worker sirviendo.
2. **Error 522 temporal en preview subdomain:** `ferra-mailer.pages.dev` mostró 522 brevemente durante el cutover. Se resolvió solo.
3. **Token Turnstile con caracteres extraños:** El TURNSTILE_SECRET (`0x4AAAAAADg7QMCY03KL8eK6ZallLLbOu-s`) tiene un guion al final que no es hex estándar, pero Cloudflare lo acepta (probablemente un separador interno).
